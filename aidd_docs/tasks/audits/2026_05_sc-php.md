# Codebase Audit for plugin/sc-php

Audit du plugin `sc-php` (32 fichiers, ~2 419 lignes Markdown, 7 skills). Le plugin est globalement cohérent avec ses pairs (`sc-python`, `sc-rust`), suit le modèle "router-based skill" et expose des skills bien délimités. Trois axes de fragilité ressortent : (1) duplication du mapping reference→target répété sur trois fichiers, (2) divergences mineures de cadre avec les plugins frères (modèle `haiku`, fichier `evals/`, skill `bruno`), (3) quelques dérives factuelles à dater (PHP 7.4 par défaut, `Strict Standards`, "PHP/Python" sur le pivot HTMX).

- Status: 🟢 Healthy with targeted fixes
- Confidence: medium-high (audit statique sur Markdown ; pas d'exécution réelle des skills)
- Scope: `plugins/sc-php/**` (skills: bruno, improve, legacy, log-analysis, setup, sniff, teach)

## Findings

- [🟡] **Duplication**: `plugins/sc-php/skills/setup/actions/01-install.md:11-23`, `plugins/sc-php/skills/sniff/actions/01-scan.md:55-67`, `plugins/sc-php/skills/sniff/actions/02-sync.md:31-45` — le tableau `Reference → Target` (6 lignes, perf+data pivots) est répété verbatim sur trois fichiers ; toute édition future devra être appliquée trois fois (extraire dans un `references/mappings.md` partagé, ou faire référencer par sniff/sync la table de `setup/01-install.md`).
- [🟡] **Duplication**: `plugins/sc-php/skills/log-analysis/actions/02-parse-errors.md:35-40`, `plugins/sc-php/skills/log-analysis/actions/03-search.md:40-41`, `plugins/sc-php/skills/log-analysis/actions/04-summarize.md:34-35` — la logique de sélection de stratégie de filtrage temporel (`docker logs --since`, Strategy A vs B) est ré-énoncée trois fois (consolider sous une seule consigne dans `references/log-formats.md` puis pointer dessus).
- [🟡] **Standards/cohérence**: `plugins/sc-php/skills/bruno/SKILL.md:3` déclare `model: haiku` alors qu'aucun pair `sc-*` n'utilise haiku ; choix légitime pour un skill simple, mais à confirmer ou aligner avec la convention `sonnet` adoptée par `sniff`/`setup` (déclarer la règle dans `MEMORY.md` projet, sinon harmoniser).
- [🟡] **Standards/cohérence**: `plugins/sc-php/skills/log-analysis/evals/scenarios.json` — fichier d'évals présent uniquement dans `sc-php` ; absent de `sc-python/skills/log-analysis/` et `sc-rust/skills/log-analysis/` (cf. `find … -name evals` retour vide). Soit propager le fichier aux deux plugins frères, soit le retirer pour éviter une garantie de couverture asymétrique.
- [🟡] **Standards/cohérence**: `plugins/sc-php/skills/setup/SKILL.md:3` et `plugins/sc-php/skills/sniff/SKILL.md:2` portent `model: sonnet` mais `plugins/sc-php/skills/improve/SKILL.md`, `legacy/SKILL.md`, `log-analysis/SKILL.md`, `teach/SKILL.md` n'ont pas de frontmatter `model:` (hérité par défaut). Décider explicitement : marquer chaque skill, ou supprimer `model:` partout pour ne dépendre que du défaut harness.
- [🟡] **Stale/factuel**: `plugins/sc-php/skills/legacy/actions/01-scan.md:18` — "If still unknown: assume PHP 7.4 and note the assumption in output". Le défaut PHP 7.4 est intentionnel : DEC-012 (SmartLockers, 2026-03-25) documente un backport vers PHP 7.4.29 sur le serveur de prod `vps621955`, migration PHP 8.x non encore finalisée. Le vrai problème est que le défaut est silencieux pour toute autre organisation — ajouter un avertissement explicite ("assumed PHP 7.4 — set `target` explicitly if your server runs a different version") plutôt que de relever la valeur.
- [🟡] **Stale/factuel**: `plugins/sc-php/skills/log-analysis/references/log-formats.md:7` — `Strict Standards` listé parmi les sévérités de log PHP. Cette sévérité a été supprimée en PHP 7.0 (fusion dans Deprecated/Notice). À retirer du tableau et de la regex ligne 9 si la cible n'est pas PHP 5.x legacy.
- [🟡] **Irrelevance/scope leak**: `plugins/sc-php/skills/setup/SKILL.md:40` et `plugins/sc-php/skills/setup/references/07-perf-pivots-htmx.md:9` qualifient le pivot HTMX de "PHP/Python backend SSR + HTMX swaps" et `sc-python/skills/setup/references/07-perf-pivots-django.md:93,98,99` y renvoie implicitement, alors que le pivot HTMX physique est uniquement dans `sc-php`. Coupling inter-plugin non documenté (déplacer le pivot HTMX dans un plugin neutre — `sc-tiers` ? — ou dupliquer une copie dans `sc-python`).
- [🟡] **Cohérence interne**: `plugins/sc-php/skills/improve/actions/01-analyze.md:8` annonce `focus` ∈ {`solid`, `patterns`, `types`, `framework`} mais le tableau de catégories Step 2 utilise les libellés "SOLID violations", "Missing patterns", "PHP type system gaps", "Framework-specific issues" ; le mapping `focus → catégorie` n'est pas explicite (ajouter une ligne `focus=solid → SOLID, focus=patterns → Missing patterns, …`).
- [🟢] **Error handling**: `plugins/sc-php/skills/log-analysis/actions/01-tail.md:36` ("If file not found: `[not found: <path>]` and continue") et `plugins/sc-php/skills/log-analysis/actions/04-summarize.md:49` ("If a source is not found or empty, mark it `[no data]`") gèrent les chemins manquants sans crash. Convention respectée.
- [🟢] **Sécurité (commande)**: `plugins/sc-php/skills/log-analysis/actions/03-search.md:36-40` documente correctement l'échappement quote-safe en SSH (`SAFE=$(printf … sed "s/'/'\\\\''/g")`) et l'ajout de `--` avant les paths pour bloquer l'interprétation pattern-as-flag. Conforme aux OWASP injection guidelines.
- [🟡] **Tests/évals manquants**: `plugins/sc-php/skills/bruno/`, `plugins/sc-php/skills/improve/`, `plugins/sc-php/skills/legacy/`, `plugins/sc-php/skills/setup/`, `plugins/sc-php/skills/sniff/`, `plugins/sc-php/skills/teach/` — aucun de ces six skills ne dispose d'un dossier `evals/` (seul `log-analysis` en a). Le critère de complétude utilisateur ("est-ce que le skill route bien ?") n'est mesurable nulle part ailleurs.
- [🟡] **Stale/factuel**: `plugins/sc-php/skills/legacy/actions/01-scan.md:57` — "Array unpacking with string keys — Not yet available — note as gap | 8.1". L'unpacking avec clés string est disponible depuis PHP 8.1 (PR officielle 2021) ; la note "Not yet available" est obsolète (mettre à jour ou retirer la ligne).
- [🟡] **Bornes manquantes**: `plugins/sc-php/skills/legacy/actions/01-scan.md:75` annonce une "Step 4 — Detect framework version gaps" mais la sortie type Step 5 (lignes 79-101) ne montre aucune ligne "Framework gaps" dans l'exemple. L'utilisateur ne voit pas ce qu'il faut afficher (ajouter au moins une ligne d'exemple type "Laravel 9 → 11 : `Route::resource(...)->only(...)` deprecated").
- [🟡] **Bornes manquantes**: `plugins/sc-php/skills/legacy/actions/02-migrate.md:10` introduit un flag `dry-run` mais ne montre pas un exemple de sortie quand `dry-run=true` (le bloc "Output" lignes 138-150 ne mentionne que les cas "Modified/New/Skipped") — ajouter une variante "Diff preview only — no writes" pour clarifier le contrat.
- [🟡] **Risque secret**: `plugins/sc-php/skills/log-analysis/references/environments.md:56` — "Never store or print the SSH key path." Bonne consigne mais aucune mention d'éviter de printer le hostname ou des éventuels tokens présents dans les logs grepés (étoffer la transversal rule en cohérence avec `log-analysis/SKILL.md:40`).
- [🟢] **Coverage**: `plugins/sc-php/skills/log-analysis/evals/scenarios.json` — 17 scénarios FR/EN couvrant les 4 actions + 2 cas négatifs (`expect_action: null`). Bonne couverture du routage.
- [🟢] **Bornes nettes**: `plugins/sc-php/skills/improve/SKILL.md:9-11` et `plugins/sc-php/skills/teach/SKILL.md:11`, `plugins/sc-php/skills/log-analysis/SKILL.md:9-11` énumèrent les "Do NOT use" vers `aidd-dev:08-debug` / `aidd-dev:05-review` / `aidd-dev:02-implement`. Les frontières inter-skills sont explicites.
- [🟢] **Cohérence interne**: les sorties type des skills `sniff`, `setup`, `legacy`, `improve` utilisent un format markdown commun (titre `📊 / 📋 / ✅` + tableau récapitulatif). Style harmonisé.

## Recommendations

Ranked by impact (high to low):

1. **Extraire le mapping reference→target dans une source unique** — créer `plugins/sc-php/skills/_shared/mappings.md` (ou équivalent intra-plugin) et y faire pointer `setup/01-install.md`, `sniff/01-scan.md`, `sniff/02-sync.md`. Éliminera la dérive de drift sur les futures additions de pivots (Inertia, Livewire, etc.).
2. **Rendre le défaut PHP explicite plutôt que silencieux** — `legacy/01-scan.md:18` : conserver PHP 7.4 comme défaut (légitimé par DEC-012 SmartLockers vps621955), mais ajouter un message d'avertissement visible ("⚠ PHP version not detected — assumed 7.4; pass `target` explicitly if your server differs"). Retirer en parallèle `Strict Standards` de `log-formats.md:7,9` (sévérité supprimée en PHP 7.0) et corriger la note "Array unpacking with string keys — Not yet available" (`legacy/01-scan.md:57`).
3. **Décider du status du skill `bruno` et du fichier `evals/`** — soit propager au reste de la famille `sc-*` (cohérence cross-plugin), soit documenter explicitement dans `README.md` que `bruno` est PHP-specific et que les évals ne sont fournies que sur `log-analysis`.
4. **Aligner les frontmatter `model:`** — choisir entre "déclarer explicitement sur chaque SKILL.md" ou "ne pas déclarer pour hériter du défaut", puis appliquer en bloc.
5. **Étendre les sorties d'exemple de `legacy/01-scan.md` et `legacy/02-migrate.md`** pour inclure (a) une section "Framework gaps" avec un exemple Laravel/Symfony, (b) une variante de sortie `dry-run`.
6. **Documenter le coupling inter-plugin HTMX** — soit déplacer `07-perf-pivots-htmx.md` vers un plugin neutre, soit l'expliciter dans le `description` du SKILL.md de `setup` ("ce pivot peut être consommé par d'autres plugins de la famille `sc-*` via web-optimize").

## Final Audit

- **Score**: 7.5 / 10 — solide cohérence d'ensemble, mais 3 patterns à harmoniser et 2 références obsolètes à dater.
- **Top risks**: (1) dérive du mapping pivots dupliqué × 3 → futur drift silencieux ; (2) défauts PHP 7.4 / `Strict Standards` qui peuvent induire des recommandations erronées en 2026 ; (3) coupling HTMX non documenté pouvant casser à un déplacement de fichier.
- **Quick wins**: ajouter le message d'avertissement PHP 7.4 assumé dans `legacy/01-scan.md:18` (≤ 15 min, voir DEC-012), corriger `legacy/01-scan.md:57` et `log-formats.md:7,9` (≤ 15 min), retirer ou propager `log-analysis/evals/scenarios.json` (≤ 1 h).
- **Follow-up actions**: créer un ticket "consolidate sc-php internal mappings" + un ticket "evals coverage pour 6 skills restants" ; décision design "skill `bruno` reste-t-il PHP-only ou devient-il `sc-tiers:bruno` ?".
- **Additional notes**: l'absence de fichier `CLAUDE.md` à la racine du repo `aidd-overlay` rend l'audit dépendant des conventions observées par parallélisme avec `sc-python` / `sc-rust` / `sc-js` ; un mémo durable des conventions cross-plugin (model frontmatter, evals, mapping refs) faciliterait les audits futurs.
