# Brief

Assembles the **`_brief/` working directory** that the `writing` plugin consumes. `brief` is the *assembler*: it consolidates the gathered inputs (concept from `forge`, lore from `lore-extract`/`research`, rules from `rules-keeper`, raw notes) plus the domain's **global/durable resources** into a **self-contained `summary.md`**, and lays out the brief structure.

## Two levels: domain `R` vs project `<projet>`

- **`R` — domain working directory.** Home of the **global/durable resources** shared across projects (lore, rules, references — typically under `R/_savoir/`), catalogued by **`R/bank.yml`** (the domain resource manifest — see `${CLAUDE_PLUGIN_ROOT}/references/bank-yml.md`). This is the *source* of durable knowledge. `R/bank.yml` is **maintained by `obsidian:tree`** (a cache, regenerated by scanning `R/_savoir/`) and **read only by `brief`, at assembly time — never by `writing`.**
- **`<projet>` — the writing project (work unit).** Where `_brief/` and `_output/` live. Typically nested under the domain at `R/<Year>/<Month>/mon-projet/`. This is `brief`'s target. **The project has no `bank.yml`** — its `summary.md` is self-contained.

`brief` **reads** global resources from `R` *at assembly time* and **consolidates them inline** into `<projet>/_brief/summary.md`. At runtime, `writing` never climbs back up to `R` — it touches only the project's `_brief/`/`_output/`. The project therefore stays **portable**: move `<projet>/` anywhere and `writing` still works, because all context is inline in `summary.md`.

> The organisation convention (the `(Perso|Pro)/category/<R=subcategory>/<Year>/<Month>/<projet>/` schema, `_`-prefixed working dirs) is owned by `obsidian:tree`. `brief` does not compute the dated path; it just needs the project dir to write into and the resources to fold in.

## Output structure

```
<projet>/_brief/
  summary.md        ← AUTONOMOUS brief: concept/synopsis + type + language
                       + consolidated lore/rules/data INLINE (incl. relevant R globals)
  personas/         ← reader personas (filled by writing:persona)
  output-styles/    ← writing styles (filled by writing:tone-finder)
```

`summary.md` is the **single source of context** for `writing` — everything needed to produce must be inline here. If an input is missing, `brief` flags it rather than pointing at an external path (including paths back into `R`).

## Available actions

| #   | Action     | Role                                                                               | Input                                                  |
| --- | ---------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------ |
| 01  | `assemble` | Create `<projet>/_brief/`, consolidate project inputs + R globals into `summary.md` | `[<projet>]` (default: CWD) `[--resources <R>]` + sources |

## Default flow

Trigger-to-action mapping:
- "build brief", "assemble brief", "prepare the brief", "nouveau brief", "monter le brief" → `assemble`
- "check brief", "is the brief ready", "vérifier le brief", "brief integrity" → `check`

## Transversal rules

- **Runnable anywhere; owns only `<projet>/`.** `brief` can be launched in any directory and is responsible **only for the contents of its project dir** (`<projet>/_brief/`). It does **not** own the global arborescence: it never creates the dated path, never manages `R`'s layout, never writes `R/bank.yml` (that is `obsidian:tree`'s job). `R` is reached only as an **explicit input** (`--resources <R>`), never auto-discovered by walking the tree.
- **Local paths only.** Target the project dir `<projet>` (argument or CWD). Global resources come from `R` (the domain, passed explicitly as `--resources <R>` per the rule above) and are read from its `R/bank.yml` manifest (or explicit source paths) — `R` is never auto-discovered by walking the tree. No global/absolute hardcoding, no `~/.jdr.yaml`. The only `bank.yml` that exists is `R/bank.yml` (domain manifest) — never one inside `<projet>/`.
- **Consolidate, don't reference.** R's globals are folded **inline** into `summary.md`. The produced `<projet>/_brief/` must be readable on its own — `writing` never reads outside it, not even back into `R`.
- `summary.md` declares **type** (technical-doc / cheat-sheet / RPG scenario / novel / guide…) and **language** (default: French) up front — `writing` selects its mode from these.
- Never overwrite existing brief files without explicit confirmation — create what is missing, propose updates for what exists.
- `personas/` and `output-styles/` are created (empty or seeded); they are filled by `writing:persona` and `writing:tone-finder`.
- `assemble` validates the assembled `summary.md` with the user before writing.

## External data

- `<projet>/_brief/summary.md` — the autonomous brief this skill produces.
- `R/bank.yml` — the domain resource manifest (`${CLAUDE_PLUGIN_ROOT}/references/bank-yml.md`): catalogues `R`'s global/durable resources so `brief` can pick what to consolidate. Read at assembly only.
- `R` (domain) global/durable resources — `R/_savoir/` lore/rules/references, consolidated inline. Source of truth lives at the domain; the project gets a self-contained copy of what it needs.
- Source files passed as arguments (concept, lore, rules, notes) — produced upstream by `forge`, `research`, `lore-extract`, `rules-keeper`.
- The `_brief/ → _output/` contract is the `writing` plugin's brief model (`writing:references/brief-model.md`). The `R` / `<projet>` layout is owned by `obsidian:tree` (`tree:references/tree-convention.md`).
