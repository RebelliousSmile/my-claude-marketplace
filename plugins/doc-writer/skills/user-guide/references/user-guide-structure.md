# User guide structure

The sections of a complete end-user guide. Include what fits the product; omit what doesn't and note the omission. Order is the default reading order.

## 1. Introduction

- One or two sentences: what the product does and what this guide covers.
- **Who this guide is for** — the reader and their assumed starting point.
- What the reader will be able to do after reading.

## 2. Before you start (prerequisites)

- Accounts, access, installs, permissions, or data the user needs first.
- Only real prerequisites — do not pad.

## 3. Getting started

- The shortest path to a first success (first value), end to end.
- Numbered steps, each one action, each stating the visible result.
- Ends on something the user can see working.

## 4. Tasks (the core)

- One subsection per user goal, titled by the goal in the user's words.
- Each task: a one-line purpose, numbered steps, expected result, and a note for common variations.
- Order tasks by frequency / logical progression, not by menu order.
- `[screenshot: …]` placeholders where a visual genuinely helps.

## 5. Tips & best practices (optional)

- Short, actionable recommendations that prevent common mistakes.

## 6. Troubleshooting

- Table or list: **symptom → likely cause → fix**.
- Cover the errors a real user actually hits, phrased as they'd describe them.

## 7. FAQ (optional)

- Real recurring questions, short answers, each linking to the relevant task section.

## 8. Glossary (optional)

- Only terms the user must understand; one-line plain definitions.

## Quality bar

- A new user can complete every documented task by following only the guide.
- No step assumes knowledge the guide hasn't given.
- Terminology matches the product UI exactly.
