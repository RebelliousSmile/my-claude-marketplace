# 01 - Generate

Create a new reader persona YAML file from a description or profile and write it to `<brief>/personas/`.

## Inputs

- `<brief>` (required, positional) — path to the brief directory; the persona is written to `<brief>/personas/`.
- `description` (required) — free-form description of the reader (e.g. `"WoT fan reader"`, `"RPG game master, experienced with ADRENALINE system"`)

## Outputs

Schema is **flat** — no nested `persona:`/`expectations:` wrapper, `criteria` values are plain weight numbers. This is the shape `review:comment` actually reads (weighted-consensus math in `review/actions/01-comment.md` step 9 needs `scope`/`weight_class` on every persona):

```yaml
# Generated by writing:persona:generate
# Version: 1.0 — 2026-XX-XX

id: "fan-wot"
name: "Fan de La Roue du Temps"
scope: project          # project | universe | global — how broadly this persona is reused
weight_class: project   # mirrors scope; consumed by review:comment's weighted consensus
                         # (project x1.0, universe x0.8, global x0.5 — review-loop.md)
description: >
  Avid reader who has read all 14 WoT volumes. Judges content against the established canon.

criteria:
  engagement: 0.25
  clarity: 0.25
  immersion: 0.30
  satisfaction: 0.20
  # Weights sum to 1.0

must_haves:
  - "Terminology consistent with the brief's lore"
  - "Magic system described faithfully (saidin/saidar distinction)"
  - "Character voices consistent with established personalities"

deal_breakers:
  - "saidin and saidar confused"
  - "Anachronistic modern vocabulary"
```

Saved to: `<brief>/personas/<id>.yaml`

## Process

1. Parse the description: identify persona archetype (fan reader, game master, casual reader, genre newcomer, etc.), domain expertise level, specific universe knowledge if mentioned.
2. Derive the persona ID: kebab-case slug from the description (e.g. `"WoT fan reader"` → `fan-wot`). Check for ID conflicts in `<brief>/personas/` (including near-collisions where a differently-worded description would slug to an existing file) — if the derived slug is already taken, pick a distinct one; never overwrite an existing persona file.
3. If `<brief>/summary.md` is present, read it to extract any universe/lore context that should inform the persona's must-haves and deal-breakers. Do not read outside `<brief>/`.
4. Determine `scope`/`weight_class`: default `project` (persona scoped to this brief only) unless the description explicitly signals reuse across projects (`universe`) or across the whole vault (`global`). This field is required — `review:comment`'s weighted consensus cannot classify a persona without it.
5. Generate the flat YAML structure (see Outputs): fill `id`/`name`/`scope`/`weight_class`/`description` plus top-level `criteria`/`must_haves`/`deal_breakers`. Derive `must_haves` from the persona's archetype and domain knowledge. Derive `deal_breakers` from what would break immersion for this reader type.
6. Infer criterion weights: weight the criterion the persona cares about most at 0.30; distribute the remaining 0.70 across the others. Weights MUST sum to 1.0.
7. Present the draft YAML to the user: "Does this persona match your vision? Anything to adjust?" Revise if needed.
8. Write to `<brief>/personas/<id>.yaml`. Never write elsewhere, even if asked (e.g. a "backup copy" request) — `<brief>/personas/` is the only valid destination (`SKILL.md` / `brief-model.md`).

## Test

After `generate <brief> "WoT fan reader"`, verify that `<brief>/personas/fan-wot.yaml` exists, uses the **flat** schema (top-level `id`, `name`, `scope`, `weight_class`, `description`, `criteria`, `must_haves`, `deal_breakers` — no nested wrapper), that criteria weights sum to exactly 1.0, and that no other file in `<brief>/personas/` was modified.
