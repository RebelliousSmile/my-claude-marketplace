# user-guide — Clarify-Before-Outline + Review-Defect-Detection + No-Fabrication Behavioural Test Scenarios

Behavioural tests for **user-guide** (`plugins/writing/skills/user-guide/SKILL.md` + `actions/01-outline.md` + `actions/02-write.md` + `actions/03-review.md` + `references/user-guide-structure.md`) — verifies that the skill (1) asks for audience/tasks in a short list rather than guessing when `outline` runs on a vague subject, (2) `review` actually catches five distinct, independently-rooted defect classes in a populated draft (banned marketing words, an undocumented-prerequisite step, a UI-titled section, a step missing its visible result, an unconfirmed feature asserted as fact) — each mapped to its own rule, (3) `review` does not invent findings against sections that already conform, and (4) `write` inserts a `[screenshot: …]` placeholder rather than fabricating UI it hasn't seen, and never asserts an unconfirmed behaviour as fact.

This suite is **distinct** from:
- `evals/scenarios.json` — a flat prompt→`expect_action` router-mapping table (does `user-guide` get invoked at all, and does it route to `outline`/`write`/`review`), no content-quality or write-scope coverage.
- **this file** — the durable regression spec for the actual documentation behaviours: clarify-before-structuring, the five-defect detection surface of `review`, `review`'s discriminance (no false positives on clean content), and `write`'s no-fabrication guarantees (screenshots and unconfirmed features).

> **Fixture / preconditions.** Run against the populated fixture **`user-guide-fixture`** (READ-ONLY), at `…/scratchpad/fixtures/writing-doc-fixtures/user-guide-fixture/`:
> - `outline.md` — the **approved** plan for "FacturePro", a web invoicing app for freelancers. Audience: freelancers with no accounting background, first contact with the tool. 8 goal-titled sections. Its "Notes / hypothèses" explicitly flag two unresolved points: (a) exact UI labels need confirming against the app/a capture, and (b) **no confirmed information on any automatic SMS payment-reminder — to be verified before mentioning it in the guide**.
> - `guide-draft.md` — a drafted guide with **five deliberate defects**, each rooted in a different rule: §3 banned marketing wording; §3 step 4 assumes a "plan de numérotation" never introduced earlier in the guide; §4 titled by a UI element ("L'écran Factures") instead of a user goal; §5 step 2 doesn't state the visible result after the click; §5 step 3 asserts an automatic SMS reminder as fact — the exact behaviour `outline.md` flagged as unconfirmed. §8 (Dépannage, a symptom→cause→fix table) and §9 (FAQ) are clean and conform to `user-guide-structure.md`.
>
> State which fixture file and which section/step is under test in every run. The judge reads the fixture but **writes nothing**; the decisive observable is each scenario's **response content** (questions asked, findings listed with their cited rule, drafted text) — reasoned out, not executed against the fixture.

## How to run

Agent-as-**user-guide** (dry-run, READ-ONLY on the fixture): load `plugins/writing/skills/user-guide/SKILL.md` + `actions/01-outline.md` + `actions/02-write.md` + `actions/03-review.md` + `references/user-guide-structure.md` + `references/output-style.md` + this suite, against the populated fixture **`user-guide-fixture`** (`outline.md` for S1/W1/W2's source-of-truth on unresolved points, `guide-draft.md` for R1-R5/DN1). For each scenario, reason out what the target **would** produce — the exact response content (clarifying questions, findings list with cited rules, drafted prose, placeholders) — and judge against the pass criteria. **Nothing is written to the fixture.**

**Decisive observables** (any violation is an automatic FAIL):
1. **Clarify before structuring** — a vague subject triggers a short numbered audience/tasks question list before any TOC appears (S1).
2. **Five-defect detection** — each of the five rooted defects (banned marketing wording, undocumented-prerequisite step, UI-titled section, missing visible result, unconfirmed feature as fact) is caught and mapped to its own rule, not merged into a vague "needs polish" note (R1-R5).
3. **No false positives** — sections that already conform (§8, §9) are not flagged (DN1).
4. **No UI fabrication** — an unseen visual becomes a `[screenshot: …]` placeholder, never invented detail (W1).
5. **No unconfirmed feature asserted as fact** — the same SMS-reminder uncertainty `outline.md` already flagged is never written as a plain step in `write`'s output (W2).

**Frictions / gaps found and closed in this pass:**
- **`01-outline.md` Inputs/Process contradiction (real gap, fixed).** The `tasks` Inputs bullet read "derive from the subject if not given," while Process step 1 said "if… the key tasks are unclear, ask in one short numbered list." On a bare subject like "FacturePro" with no description/access, an implementation reading Inputs first could silently invent a task list instead of asking, defeating S1. Fixed by qualifying the Inputs bullet: derive only when the subject already carries enough material (access/description); otherwise ask, per step 1.
- **`03-review.md` truthfulness check had no explicit tie to the outline's own assumptions note (real gap, fixed).** Step 6 said "no invented features/behavior" but review's Inputs never listed the approved outline as a cross-check source, so R5's detection rested on general judgment rather than a citable instruction. Fixed by adding the outline (with its assumptions note) to review's Inputs, and by naming the cross-check explicitly inside step 6.
- **`03-review.md` had no explicit anti-fabrication rule for its own findings (real gap, fixed).** DN1 (no false positives on clean §8/§9) held only by the absence of a violated criterion, not by an instruction forbidding invented findings — the same asymmetry `01-outline`/`02-write` already close for guide content ("flag gaps rather than guess," "never fabricate UI"). Fixed by adding a closing Process line: only raise a finding where a check is actually violated.
- **No fix needed:** R1's citation match relies on `references/output-style.md`'s French banned-word list ("puissant") rather than `03-review.md`'s own (English-example) list — the generic wording "no marketing words" in `03-review.md` covers this by design (examples are illustrative, not exhaustive); not a gap.

**Tally:** 9/9 PASS (0 N/A, 0 FAIL). First run for this suite — no regression baseline. Two real spec gaps found during judging (outline's derive-vs-ask contradiction; review's ungrounded truthfulness cross-check and missing anti-fabrication rule) were closed in `actions/01-outline.md` and `actions/03-review.md` before finalizing this tally.
