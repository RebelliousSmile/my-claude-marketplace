# Brief

Assembles the **`_brief/` working directory** that the `writing` plugin consumes. `brief` is the *assembler*: it consolidates the gathered inputs (concept from `writing:forge`, data from `obs:research`, raw notes — plus, when the domain is a JDR game, lore from `ttrpg:lore-extract` and rules from `ttrpg:rules-keeper`) plus the domain's **durable resources** into a **self-contained `summary.md`**, and lays out the brief structure.

## Two levels: domain `R` vs project `<projet>`

- **`R` — domain working directory.** Home of the **durable resources** shared across projects. They live in **working-dir buckets `R/_<bucket>/`** (bucket names are free) and are **catalogued by `R/bank.yml`** (the domain resource manifest — see `${CLAUDE_PLUGIN_ROOT}/references/bank-yml.md`). `brief` consolidates **what `R/bank.yml` catalogues**, whatever the buckets are named. *(In the JDR profile the buckets are `R/_univers/`, `R/_systeme/`, `R/_subsystems/`; another domain — a novel, a knowledge base — names them differently. See `${CLAUDE_PLUGIN_ROOT}/references/domain-layout.md` for the generic model (§ JDR profile for the JDR profile).)* This is the *source* of durable knowledge. `R/bank.yml` is **maintained by `obs:tree`** (a cache, regenerated by scanning `R`'s working dirs) and **read only by `brief`, at assembly time — never by `writing`.**
- **`<projet>` — the writing project (work unit).** Where `_brief/` and `_output/` live. Typically nested under the domain at `R/<Year>/<Month>/mon-projet/`. This is `brief`'s target. **The project has no `bank.yml`** — its `summary.md` is self-contained.

`brief` **reads** global resources from `R` *at assembly time* and **consolidates them inline** into `<projet>/_brief/summary.md`. At runtime, `writing` never climbs back up to `R` — it touches only the project's `_brief/`/`_output/`. The project therefore stays **portable**: move `<projet>/` anywhere and `writing` still works, because all context is inline in `summary.md`.

> The organisation convention (the `(Perso|Pro)/category/<R=subcategory>/<Year>/<Month>/<projet>/` schema, `_`-prefixed working dirs) is owned by `obs:tree`. `brief` does not compute the dated path; it just needs the project dir to write into and the resources to fold in.

## Output structure

```
<projet>/_brief/
  summary.md        ← AUTONOMOUS brief: concept/synopsis + type + language
                       + consolidated lore/rules/data INLINE (incl. relevant R globals)
  personas/         ← ≥3 distinct reader personas (filled by writing:persona)
  output-styles/    ← ≥3 distinct writing styles (filled by writing:tone-finder)
```

`summary.md` is the **single source of context** for `writing` — everything needed to produce must be inline here. If an input is missing, `brief` flags it rather than pointing at an external path (including paths back into `R`).

## Available actions

| #   | Action     | Role                                                                               | Input                                                  |
| --- | ---------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------ |
| 01  | `assemble` | Create `<projet>/_brief/`, consolidate project inputs + R globals into `summary.md` | `[<projet>]` (default: CWD) `[--resources <R>]` + sources |

## Default flow

Trigger-to-action mapping:
- "build brief", "assemble brief", "prepare the brief", "nouveau brief", "monter le brief" → `assemble`
- "check brief", "is the brief ready", "vérifier le brief", "brief integrity" → `check`

## Transversal rules

- **Runnable anywhere; owns only `<projet>/`.** `brief` can be launched in any directory and is responsible **only for the contents of its project dir** (`<projet>/_brief/`). It does **not** own the global arborescence: it never creates the dated path, never manages `R`'s layout, never writes `R/bank.yml` (that is `obs:tree`'s job). `R` is reached only as an **explicit input** (`--resources <R>`), never auto-discovered by walking the tree.
- **Local paths only.** Target the project dir `<projet>` (argument or CWD). Global resources come from `R` (the domain, passed explicitly as `--resources <R>` per the rule above) and are read from its `R/bank.yml` manifest (or explicit source paths) — `R` is never auto-discovered by walking the tree. No global/absolute hardcoding, no `~/.jdr.yaml`. The only `bank.yml` that exists is `R/bank.yml` (domain manifest) — never one inside `<projet>/`.
- **Consolidate, don't reference.** R's globals are folded **inline** into `summary.md`. The produced `<projet>/_brief/` must be readable on its own — `writing` never reads outside it, not even back into `R`.
- `summary.md` **opens with a YAML front-matter** declaring `type:` (technical-doc / cheat-sheet / rpg-scenario / novel / guide…) and `language:` (default: `fr`) — both keys are **mandatory** (declared up front, machine-checkable). `writing` selects its mode from these.
- Never overwrite existing brief files without explicit confirmation — create what is missing, propose updates for what exists.
- `personas/` and `output-styles/` are seeded with **at least 3 distinct entries each** (filled by `writing:persona` / `writing:tone-finder`); a brief with fewer is flagged incomplete by `check` (contract: `writing:references/brief-model.md`).
- `assemble` validates the assembled `summary.md` with the user before writing.

## External data

- `<projet>/_brief/summary.md` — the autonomous brief this skill produces.
- `R/bank.yml` — the domain resource manifest (`${CLAUDE_PLUGIN_ROOT}/references/bank-yml.md`): catalogues `R`'s global/durable resources so `brief` can pick what to consolidate. Read at assembly only.
- `R` (domain) durable resources — **whatever `R/bank.yml` catalogues** (the buckets `R/_<bucket>/`, whatever their names), consolidated inline. Source of truth lives at the domain; the project gets a self-contained copy of what it needs. *(JDR profile buckets: `_univers`/`_systeme`/`_subsystems`.)*
- Source files passed as arguments (concept, data, notes…) — produced upstream by the generic feeders `writing:forge` (concept) and `obs:research` (data); when the domain is JDR, also `ttrpg:lore-extract` (lore) and `ttrpg:rules-keeper` (rules).
- The `_brief/ → _output/` contract is the `writing` plugin's brief model (`writing:references/brief-model.md`). The generic `R` / `<projet>` layout convention is `${CLAUDE_PLUGIN_ROOT}/references/domain-layout.md` (with the JDR profile documented in its § JDR profile); the dated arborescence itself is owned by `obs:tree` (`obs:references/tree-convention.md`).
